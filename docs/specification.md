# DayNavi 要件定義書

## 1. システム概要

DayNaviは、日々の予定、タスク、就職活動のイベントを一元管理するWebアプリケーションである。

ユーザーは予定を登録し、今日の予定や日付ごとの予定を確認できる。また、予定にカテゴリ、開始時刻、終了時刻、確認時刻、メモ、完了状態を設定できる。

Version 1では、予定管理の基本機能とユーザーごとのデータ管理を実装する。Version 2以降では、メールによるパスワード再設定、指定時刻での今日の予定リマインド、月間カレンダー表示などを追加する。Version 3以降では、AIによる予定・タスク作成補助を追加する。

## 3. 機能要件

### 3.1 ユーザー管理機能

- ユーザーはアカウント登録できる
- ユーザーはログインできる
- ユーザーはログアウトできる
- ログインしていないユーザーは予定を登録・編集・削除できない
- ログインしていないユーザーは自分用の予定一覧を閲覧できない
- ユーザーごとに予定データを分離する

### 3.2 パスワード再設定機能

Version 2で実装する。

- ユーザーはパスワードを忘れた場合、登録済みメールアドレスを入力できる
- システムは入力されたメールアドレス宛にパスワード再設定用リンクを送信する
- ユーザーは再設定用リンクから新しいパスワードを設定できる
- 再設定用リンクには有効期限を設ける
- 開発初期ではDjangoのコンソールメールバックエンドを使用して動作確認する

### 3.3 予定管理機能

- ユーザーは予定を登録できる
- ユーザーは自分が登録した予定のみ閲覧できる
- ユーザーは予定の詳細を確認できる
- ユーザーは予定を編集できる
- ユーザーは予定を削除できる
- ユーザーは予定の完了状態を変更できる

予定には以下の情報を登録できる。

- タイトル
- カテゴリ
- 日付
- 開始時刻
- 終了時刻
- 確認時刻
- メモ
- 完了状態

### 3.4 今日の予定表示機能

- ユーザーは今日の予定を一覧で確認できる
- 今日の日付に該当する予定のみを表示する
- 予定は開始時刻順に表示する
- 開始時刻が未設定の予定も表示する
- 完了済みの予定と未完了の予定を区別して表示する

### 3.5 日付ごとの予定表示機能

- ユーザーは日付を指定して予定を確認できる
- 指定した日付に該当する予定のみを表示する
- 日付ごとの予定は開始時刻順に表示する

### 3.6 カテゴリ管理機能

予定には以下のカテゴリを設定できる。

- 就活
- 学習
- 開発
- 大学
- アルバイト
- 個人予定
- その他

### 3.7 確認時刻機能

- ユーザーは予定ごとに確認時刻を設定できる
- 確認時刻が設定された予定を画面上で確認できる
- Version 1では実際の外部通知は行わず、アプリ内表示を対象とする

### 3.8 デイリーリマインド設定機能

- ユーザーは今日の予定をまとめて確認したい時刻を設定できる
- Version 1ではリマインド時刻を保存できる状態を目標とする
- Version 1では指定時刻での自動メール通知やバックグラウンド処理は実装しない
- Version 2以降で、設定時刻に今日の予定をメールなどで通知する

### 3.9 検索・絞り込み機能

Version 2で実装する。

- ユーザーは予定タイトルで検索できる
- ユーザーはカテゴリで予定を絞り込みできる
- ユーザーは未完了の予定のみ表示できる
- ユーザーは締切や日付が近い予定を確認できる

### 3.10 AI予定・タスク作成補助機能

Version 3で実装する。

- ユーザーは自然文で予定やタスクの内容を入力できる
- システムは入力文から、予定名、日付、開始時刻、終了時刻、カテゴリ、確認時刻、メモを抽出する
- システムは必要に応じて、関連するタスク案を生成する
- AIが生成した内容は、登録前に確認画面でユーザーに表示する
- ユーザーはAIが生成した内容を修正できる
- ユーザーが確定した場合のみ、予定またはタスクとして登録する

## 4. 非機能要件

### 4.1 セキュリティ

- ユーザーごとに予定データを分離する
- 他ユーザーの予定を閲覧・編集・削除できないようにする
- パスワード管理にはDjango標準の認証機能を使用する
- APIキー、メール認証情報、シークレットキーをGitHubに公開しない
- 本物の就活予定や個人情報をサンプルデータとしてGitHubに含めない

### 4.2 操作性

- 初めて使う人でも予定登録しやすい画面にする
- 今日の予定をトップページまたは専用画面で確認しやすくする
- 予定の登録、詳細、編集、削除への導線を分かりやすくする
- カテゴリを見て予定の種類が分かるようにする

### 4.3 保守性

- accounts、schedules など機能ごとにDjangoアプリを分ける
- モデル、ビュー、フォーム、テンプレートを分離して実装する
- READMEとdocsに設計内容を残す
- Versionごとに実装範囲を分ける

### 4.4 拡張性

- 将来的にメール通知、AI機能、Google Calendar連携を追加しやすい構成にする
- 予定データとユーザー設定データを分けて管理できるようにする
- AIの提案内容を直接保存せず、ユーザー確認後に登録する設計にする

## 5. 画面要件

Version 1で作成する画面は以下の通りである。

| 画面名 | 概要 |
|---|---|
| トップページ | アプリ概要または今日の予定への導線を表示する |
| ユーザー登録画面 | 新規ユーザー登録を行う |
| ログイン画面 | ユーザーがログインする |
| 予定一覧画面 | 自分の予定を一覧表示する |
| 今日の予定画面 | 今日の予定を一覧表示する |
| 日付別予定画面 | 指定した日付の予定を表示する |
| 予定詳細画面 | 予定の詳細を表示する |
| 予定登録画面 | 新しい予定を登録する |
| 予定編集画面 | 既存の予定を編集する |
| 予定削除確認画面 | 予定削除前に確認する |
| リマインド設定画面 | 今日の予定を確認したい時刻を設定する |

Version 2以降で追加する画面は以下の通りである。

| 画面名 | 概要 |
|---|---|
| パスワード再設定画面 | メールアドレスを入力して再設定メールを送る |
| 月間カレンダー画面 | 月単位で予定を表示する |
| 検索・絞り込み画面 | 条件に応じて予定を検索する |

Version 3以降で追加する画面は以下の通りである。

| 画面名 | 概要 |
|---|---|
| AI予定作成画面 | 自然文から予定やタスク案を作成する |
| AI提案確認画面 | AIが作成した予定案を確認・修正する |

## 6. データ要件

### 6.1 User

Django標準のUserモデルを使用する。

主な項目は以下の通りである。

- username
- email
- password

### 6.2 Schedule

予定情報を管理する。

主な項目は以下の通りである。

- user
- title
- category
- date
- start_time
- end_time
- reminder_time
- memo
- is_done
- created_at
- updated_at

### 6.3 UserReminderSetting

ユーザーごとのデイリーリマインド設定を管理する。

主な項目は以下の通りである。

- user
- daily_reminder_enabled
- daily_reminder_time
- reminder_method
- created_at
- updated_at

Version 1では、アプリ内表示または設定値の保存を対象とし、外部通知は行わない。

### 6.4 AiSuggestion

AIが生成した予定・タスク案を管理する。

主な項目は以下の通りである。

- user
- input_text
- suggested_data
- is_applied
- created_at

Version 3で実装する。
